The Blood Tracking app has been successfully enhanced with a new BloodUnit model and comprehensive API endpoints for managing blood units throughout their complete lifecycle.

## What Was Built

### 1. New Database Model: BloodUnit ✅

**Purpose:** Track individual blood units with complete lifecycle management.

**Fields:**
- `unit_id` - Unique UUID field (autogenerated)
- `donor` - ForeignKey to Donor model
- `blood_type` - CharField with 8 blood type choices
- `collection_date` - DateTimeField when blood was collected
- `expiry_date` - DateTimeField when blood expires
- `current_location` - ForeignKey to Hospital (nullable)
- `status` - CharField with 5 status choices
- `hiv_test` - BooleanField for HIV test result
- `hepatitis_test` - BooleanField for Hepatitis test result
- `blockchain_tx_hash` - CharField to store Ethereum transaction hash
- `status_history` - JSONField to track all status changes
- `created_at` - Auto timestamp on creation
- `updated_at` - Auto timestamp on updates

**Features:**
- Automatic status history tracking
- UUID for unique identification
- 4 database indexes for query optimization
- Related name for easy reverse lookups

### 2. Serializers (4 Classes) ✅

**BloodUnitSerializer (Basic)**
- Used for list endpoints
- Includes donor name and hospital name
- Readonly computed fields

**BloodUnitDetailSerializer (Full)**
- Used for detail endpoints and lifecycle history
- Includes complete nested donor and hospital info
- Lifecycle summary with status changes
- Expiry tracking and analysis

**BloodUnitCreateSerializer (Create-only)**
- Simplified for POST requests
- Validates expiry_date > collection_date
- Prevents invalid data entry

**BloodUnitStatusUpdateSerializer (Update-only)**
- Specialized for status updates
- Allows updating: status, current_location, blockchain_tx_hash

### 3. ViewSet with 6+ Endpoints ✅

**BloodUnitViewSet Features:**

1. **Create** - POST new blood unit
   - Full validation
   - Auto-generates UUID

2. **List** - GET all blood units
   - Search by unit_id, donor name, blood type
   - Filter and ordering
   - Pagination (10 items/page)

3. **Retrieve** - GET single unit
   - Full lifecycle history included
   - Complete donor and hospital info
   - Status change tracking

4. **Update/Partial Update** - PATCH blood unit
   - Update any field
   - Automatic timestamp

5. **Delete** - DELETE blood unit
   - Remove from system

**Custom Actions:**

6. **by_blood_type** - `/units/by_blood_type/`
   - Required: blood_type parameter
   - Returns: Only units in storage
   - Use case: Find available blood quickly

7. **lifecycle_history** - `/units/{id}/lifecycle_history/`
   - Returns: Complete lifecycle of unit
   - Includes: All status changes with timestamps and notes
   - Use case: Full unit audit trail

8. **update_status** - `/units/{id}/update_status/`
   - Updates: status with automatic history recording
   - Records: Previous status, timestamp, notes
   - Use case: Transition unit through lifecycle

9. **available_units** - `/units/available_units/`
   - Returns: All units currently in storage
   - Use case: Inventory check

10. **units_at_location** - `/units/units_at_location/`
    - Parameter: location_id (Hospital ID)
    - Use case: Location-specific inventory

11. **near_expiry** - `/units/near_expiry/`
    - Returns: Units expiring in 7 days
    - Ordered by: Closest expiry first
    - Use case: Urgent inventory management

## Technical Specifications

### Database Schema

**Indexes Created:**
- `blood_type` - For blood type filtering
- `status` - For status-based queries
- `(donor_id, collection_date)` - For donor-specific queries
- `(current_location_id, status)` - For location inventory

**Relationships:**
- BloodUnit.donor → Donor (ForeignKey, required)
- BloodUnit.current_location → Hospital (ForeignKey, optional)

### API Specifications

**Authentication:**
- Token-based (DRF TokenAuthentication)
- Required on all endpoints

**Permissions:**
- IsAuthenticated on all endpoints
- No role-based restrictions (can be added)

**Response Format:**
- JSON
- Consistent error handling
- HTTP status codes: 200, 201, 400, 401, 404, 500

**Pagination:**
- 10 items per page
- Page navigation via `?page=N`

**Filtering Options:**
- by_blood_type (required parameter)
- by_status (via list endpoint)
- by_location (via units_at_location action)
- SearchFilter (unit_id, donor name, blood type)
- OrderingFilter (collection_date, expiry_date)

### Data Validation

**At Creation:**
- Donor must exist
- Blood type must be valid choice
- Expiry date must be after collection date
- Collection and expiry dates are required

**At Status Update:**
- Status must be valid choice
- When status = 'transfused', fulfilled_at timestamp set
- When status = 'expired', system tracks automatically

## Usage Examples

### Create Blood Unit
```bash
curl -X POST http://localhost:8000/api/blood-tracking/units/ \
  -H "Authorization: Token YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "donor": 1,
    "blood_type": "O+",
    "collection_date": "2026-04-02T10:30:00Z",
    "expiry_date": "2026-05-02T10:30:00Z",
    "hiv_test": false,
    "hepatitis_test": false
  }'
```

### List O+ Units
```bash
curl -X GET "http://localhost:8000/api/blood-tracking/units/by_blood_type/?blood_type=O+" \
  -H "Authorization: Token YOUR_TOKEN"
```

### Update Status
```bash
curl -X PATCH "http://localhost:8000/api/blood-tracking/units/1/update_status/" \
  -H "Authorization: Token YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "status": "storage",
    "current_location": 1,
    "notes": "Tests passed"
  }'
```

### Get Lifecycle
```bash
curl -X GET "http://localhost:8000/api/blood-tracking/units/1/lifecycle_history/" \
  -H "Authorization: Token YOUR_TOKEN"
```

## Status Flow

```
collected → testing → storage → transfused
                   ↓
                 expired
         (automatic after expiry_date)
```

**Descriptions:**
- **collected**: Unit just drawn from donor
- **testing**: Unit undergoing lab tests
- **storage**: Tests passed, in cold storage
- **transfused**: Unit given to patient
- **expired**: Unit past expiry date

- Expiry date must be after collection date

- 401: Authentication required
- 404: Unit not found
